## Phases

### Phase 18: Member ZT Client Version

**Goal:** Network operators can see, at a glance, which ZeroTier client version each member is running alongside its online status — and never see `undefined`/`null`/spinner states for members the controller has not reported a version for.
**Depends on:** Nothing (first phase of v3.1)
**Requirements:** MEMBER-01, MEMBER-02
**Plans:** 2 plans

Plans:
- [x] 18-01: Types + memberService.listMembersWithPeers + service tests (Wave 1) — completed 2026-05-11
- [x] 18-02: network-detail Status column version sub-line + page tests (Wave 2) — completed 2026-05-11

**Details:**

Service-layer slice introduced `MemberWithPeer extends Member { version?: string }` and `MemberWithPeerListResponse` view types (src/types/zerotier.ts:208-220), and a new `memberService.listMembersWithPeers(networkId)` method that runs `listMembers` + `nodeService.getPeers()` in parallel via `Promise.all`, merges by nodeId, applies the locked D-07 filter (`/^0\.0\.0(\.|$)/` strips uninstalled-client placeholders), and degrades gracefully on peer-fetch failure (`.catch(() => [])`). The page slice rewrote the Status column to render the existing `<zt-badge variant="success">Online</zt-badge>` followed by a `<span class="version">· vX.Y.Z</span>` sub-line, widened the column from 90px to 180px, and locked three render states (online+known, online+unknown shows `· —`, offline absent) with new B-UI-1..B-UI-4 tests.

## Milestone Summary

**Key Decisions (locked, with outcomes):**

- D-09 (Phase 18, ✓ Good): `version` flows via new `MemberWithPeer` view type extending `Member` rather than mutating the canonical `Member` interface. Keeps single-getter semantics intact and avoids type churn in unrelated callers.
- D-13 (Phase 18, ✓ Good): client-side merge of `/peer` + `/member` results; no new backend route. Graceful degrade via `.catch(() => [])` on the peer leg.
- D-03 (Phase 19, ✓ Good with one amendment): two-pass grep audit with 26-char accent class + 43-token list. Amendment 2026-05-11 changed locale prefix `LC_ALL=C` → `LC_ALL=C.UTF-8` to avoid GNU grep 3.11 byte-class overlap with Unicode punctuation (em-dash, ellipsis, checkmarks produced 94 false positives under plain `LC_ALL=C`).
- D-06 (Phase 19, ✓ Good): `audit:i18n` is opt-in only — never wired into `prepare`/`preinstall`/`postinstall`/`precommit`/`pretest`/`test`/`lint`/`build`. Developer-tooling pattern preserved.
- D-04 override (Phase 19, ⚠️ Revisit): row-by-row walkthrough discipline overridden via `/gsd-next --force`; reviewer self-attested. Acceptable because the automated audit script's exit 0 already covered the evidence axis, but the override should not become routine.
- Plan 20-03 (Phase 20, ✓ Good): the persistent `<zt-navbar>` consumes route metadata via `(e as CustomEvent).detail?.location?.route` inside the existing `vaadin-router-location-changed` listener — NOT `Router.location`, which would have shipped silently broken (Vaadin Router 2.x exposes `location` only as an instance property; static access returns undefined). The synthetic-event regression test in 20-03 Task 3 Test 4 is the runtime guard.
- Option B / IN-04 (Phase 20, ✓ Good — gap closure): `<zt-app>.theme` is the single source of truth; navbar's `currentTheme` is a parent-bound `@property` consumed via `.currentTheme=${this.theme}` Lit binding. Eliminates the multi-writer anti-pattern that produced CR-01. Opt-out persistence via `options.persist?: boolean` preserves the pre-Phase-20 "System" UX contract (live UI update without localStorage poisoning).
- LAYOUT-02 fix path (Phase 20, ⚠️ Revisit): the byte-for-byte "geometry mirror" approach (navbar :host duplicates sidebar `.brand` padding/border) proved fragile — two follow-up fixes were needed to close the alignment gap (8f622f1 stacked-content height + f1aa201 .nav-title-stack height-lock + f979081 :host padding tune). Future header-alignment work should consider a shared `--header-band-padding` token rather than re-deriving values per-side.
