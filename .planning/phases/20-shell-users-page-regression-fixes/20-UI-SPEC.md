# Phase 20 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Phase character:** regression fix, not redesign. Every token and copy string referenced here already exists in the codebase; this contract pins down which existing values Phase 20's three deliverables must reuse, and the few new strings (route-metadata `title`/`subtitle`) that the planner derives from current per-page `<zt-navbar>` invocations.

---

## Design System

| Property          | Value                                                                                                                  |
|-------------------|------------------------------------------------------------------------------------------------------------------------|
| Tool              | none (Lit + custom CSS custom-property token system; no shadcn, no Tailwind)                                           |
| Preset            | not applicable                                                                                                          |
| Component library | none external — internal web components under `src/components/` (`<zt-sidebar>`, `<zt-navbar>`, `<zt-data-table>`, …) |
| Icon library      | `lucide-static` (locked by REQUIREMENTS.md Out-of-Scope and v3.0 D-button-system; Phase 20 must use it)                |
| Font              | `Inter, -apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif` (token: `--font-family` in `src/styles/theme.ts:32`) |

**shadcn initialization gate:** Skipped. The project is deliberately framework-light (Lit + Vite, no Tailwind/shadcn) and v3.0 already locked the `.btn-*` + CSS-custom-property token system as the standardization target. Introducing shadcn in a polish phase would contradict SC #4 ("`.btn-*` system remains the standardization target — icons are added inside the existing button classes, not via bespoke styling"). Source: `.planning/PROJECT.md` § Key Decisions; ROADMAP.md § Phase 20 SC #4.

## Typography

Phase 20 uses exactly 4 of the 7 declared `--font-size-*` tokens. Weights stay at 2 values (regular `500` for body/labels, semibold `600` for titles). Line heights are inherited from browser defaults except where Lit components override (existing `.btn` uses `line-height: 1.4` at `shared.ts:41` — preserved).

| Role                           | Size                          | Weight | Line Height       | Token / source                                                          |
|--------------------------------|-------------------------------|--------|-------------------|-------------------------------------------------------------------------|
| Navbar title (route-derived)   | `1rem` (16px) — `--font-size-lg` | 600    | default (~1.5)    | `src/components/navbar.ts:41-44` — preserved verbatim                   |
| Navbar subtitle (route-derived)| `0.75rem` (12px) — `--font-size-xs` | regular | default       | `src/components/navbar.ts:46-49` — preserved verbatim                   |
| Action-button hover/disabled label (`aria-label`, `title`) | n/a (assistive text, not rendered as visible text) | — | — | `src/pages/users.ts:144,153,162` — preserved verbatim |
| Status indicator label ("Connected" / "Disconnected") | `0.75rem` — `--font-size-xs` | regular | default | `src/components/navbar.ts:100-102` — preserved verbatim                 |

**Phase 20 introduces no new typographic decisions.** Existing components (`<zt-navbar>`, `<zt-data-table>`, `<zt-users-page>`) already declare their type stack via `sharedStyles` + their own `static styles`. The contract is to leave these untouched in the existing components and only inherit them via the new `<zt-data-table>` `sharedStyles` import (D-01).

**Phase 19 i18n constraint (carried forward):** All new strings written into `src/router/index.ts` route metadata (`title`, `subtitle`) MUST be English. Phase 19's `npm run audit:i18n` script will re-flag any PT regression on next CI run.

## Color

Existing tokens in `src/styles/theme.ts` (dark theme defaults `:host`, lines 4-29; light theme override `:host([theme="light"])`, lines 75-101). Phase 20 introduces zero new color literals.

| Role             | Value (dark)              | Value (light)             | Usage in Phase 20                                                              |
|------------------|---------------------------|---------------------------|--------------------------------------------------------------------------------|
| Dominant (60%)   | `#0f1117` (`--color-bg-primary`) | `#f8f9fc`         | App background; main content area background                                   |
| Secondary (30%)  | `#161923` (`--color-bg-secondary`) | `#ffffff`       | Sidebar background, navbar background (`var(--color-bg-secondary)`), card surfaces |
| Accent (10%)     | `#F5A623` (`--color-accent`) | `#9A6500`              | See "Accent reserved for" below                                                |
| Destructive      | `#ef4444` (`--color-error`) | `#c91f1f`               | Used by `.btn-danger` for the delete-user confirmation modal action; Phase 20 does NOT touch destructive copy |

**Accent reserved for** (explicit list — Phase 20 must NOT extend accent to other elements):

- Active sidebar nav link (`.nav-link.active`: `color`, `background: var(--color-accent-muted)`, `border-left: 3px solid var(--color-accent)` — `src/components/sidebar.ts:84-88`)
- Primary CTA button background (`.btn-primary` — `src/styles/shared.ts:49-55`)
- Input focus border (`.input:focus` — `src/styles/shared.ts:132-134`)
- Loading spinner top arc (`.spinner` border-top — `src/styles/shared.ts:265-272`)

**Phase 20 accent contract:** The persistent navbar's title, subtitle, log-out icon, theme-toggle icon, and connection-indicator MUST NOT use `--color-accent`. They use `--color-text-primary` / `--color-text-secondary` / `--color-text-muted` / `--color-success` / `--color-error` (already in `navbar.ts`). Verified: navbar component template preserves the current color palette; no new accent surfaces are introduced.

**Action-button (Pencil/KeyRound/Trash2) color contract:**

- Default state: `color: var(--color-text-secondary)` (≈ `#9498b0` dark, `#5a5e72` light) — inherited from `.btn-icon` at `shared.ts:87`
- Hover state: `color: var(--color-text-primary)`, `background: var(--color-bg-hover)` — `shared.ts:92-95`
- Disabled state: `opacity: 0.3`, `color: var(--color-text-muted)` — `shared.ts:97-101`
- The Trash2 (delete) icon stays the same neutral color as Pencil/KeyRound in idle state — destructive intent is communicated by the confirmation modal, not by tinting the trigger icon red. This matches existing convention across the app (see `network-detail.ts` member-delete trigger).

## Interaction Contract (states)

For each interactive element Phase 20 touches, declare every state the executor must implement:

### Users-page action buttons (Edit / Reset / Delete)

| State                           | Visual contract                                                                                                       |
|---------------------------------|-----------------------------------------------------------------------------------------------------------------------|
| Idle                            | Icon at 16×16px, color `var(--color-text-secondary)`, background transparent, 0.4rem padding (the `.btn-icon` defaults) |
| Hover (enabled)                 | Color `var(--color-text-primary)`, background `var(--color-bg-hover)` (`shared.ts:92-95`)                            |
| Focus (keyboard)                | Browser default focus ring (no override); cursor pointer                                                              |
| Disabled (self / last admin)    | `opacity: 0.3`, `color: var(--color-text-muted)`, `cursor: not-allowed`; click handler still bound but guarded (`shared.ts:97-101` + users.ts pattern) |
| Click (enabled, Edit)           | Opens edit modal (`openEditModal(user)`)                                                                              |
| Click (enabled, Reset)          | Opens reset-password modal (`openResetModal(user)`)                                                                   |
| Click (enabled, Delete)         | Opens delete-confirmation modal (`openDeleteModal(user)`) — confirmation modal copy unchanged                         |

### Persistent navbar (lives in `<zt-app>`)

| State                                                   | Visual contract                                                                                                   |
|---------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------|
| Authenticated route (`/dashboard`, `/networks`, …)      | Rendered above `<div id="outlet">`; title + subtitle from route metadata; sticky/fixed at top of `.main-content`; height matches `.brand` row exactly |
| Unauthenticated route (`/login`, `/setup`)              | NOT rendered (CONTEXT D-06)                                                                                       |
| Route change (e.g., `/dashboard` → `/networks`)         | Component instance persists; title/subtitle re-render from new route metadata; connection-indicator state and theme-toggle state survive (no re-mount; no flicker) |
| Theme toggle                                            | Existing behavior preserved — clicking sun/moon icon calls `app.toggleTheme()` and re-renders icon                |
| Log-out                                                 | Existing behavior preserved — `_handleLogout()` calls `DELETE /api/auth/logout`, clears `userService`, redirects to `/login` |
| Connection indicator (Connected / Disconnected)         | Existing 30-second polling preserved; dot+label render unchanged                                                  |
| Connection indicator (checking)                         | `pulse` animation on the dot for the duration of the in-flight request; existing behavior preserved               |

### Unified header band (LAYOUT-02 visual outcome)

| Visual rule                                                                                                          | Verification path                                                              |
|----------------------------------------------------------------------------------------------------------------------|--------------------------------------------------------------------------------|
| Navbar top edge aligns with viewport top (y = 0)                                                                     | `position: fixed; top: 0` OR equivalent sticky positioning at top of `.main-content` |
| Navbar bottom edge aligns with `.brand` row bottom edge across the sidebar→main boundary                             | Identical padding-block (`1rem`) + identical 1px bottom border on both          |
| No visible gap between navbar bottom border and the page content below                                               | Page content sits directly below the 1px border (no spacer / margin-top)        |
| No visible offset / shift when navigating between routes                                                             | Navbar instance persists across route changes (no remount = no layout thrash)   |
