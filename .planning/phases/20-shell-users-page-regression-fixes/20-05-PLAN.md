<objective>
Close the CR-01 BLOCKER from `20-VERIFICATION.md`: toggling theme from the Settings page leaves the persistent navbar's sun/moon icon and aria-label stale because (a) `settings.ts setTheme()` bypasses `<zt-app>.toggleTheme()`, (b) `<zt-navbar>.currentTheme` is an `@state` that reads localStorage only at `connectedCallback`, and (c) `src/app.ts` does not thread `theme` down to the navbar. Phase 20's persistent-navbar refactor (Plan 20-03) removed the per-page remount that previously masked this two-way-binding gap.

The fix adopts **Option B (IN-04)** — single source of truth via parent → child property binding. This makes CR-01 impossible by construction:

1. `<zt-navbar>.currentTheme` becomes a `@property` (parent-bound) instead of `@state` (self-managed). The localStorage read in `connectedCallback` is removed. The navbar can no longer get out of sync with the app's theme — it has no independent state to drift. Lit's property-binding semantics assign properties BEFORE `connectedCallback` runs, so the navbar receives its first `currentTheme` value during the parent's first render — both at app boot and at login → authenticated re-mount. (See I-1 note.)
2. `<zt-app>` adds `.currentTheme=${this.theme}` to its `<zt-navbar>` render binding, threading the app's `@state`-tracked theme down. When `app.theme` changes, Lit re-renders the navbar with the new `currentTheme` property → icon + aria-label + title update automatically.
3. `<zt-app>` gains a new public `setTheme(target: 'dark' | 'light', options?: { persist?: boolean })` method that mutates `this.theme` (the `@state`), syncs the `theme` attribute, and conditionally writes localStorage. `options.persist` defaults to `true` (the normal explicit-theme path), but the Settings page's 'system' branch passes `false` so it can preserve the pre-Phase-20 "no localStorage key = follow OS preference on next boot" contract. `<zt-app>.toggleTheme()` is refactored to delegate to `setTheme(target)` (persist defaults to true) for consistency.
4. `src/pages/settings.ts setTheme()` is rewritten to resolve 'system' → 'dark' | 'light' at the call site (preserving the existing UX), then call `app.setTheme(resolved, { persist: false })` for the 'system' branch (followed by `localStorage.removeItem('zt-theme')` to clear the key for next-boot OS-preference re-read) or `app.setTheme(resolved)` for the explicit dark/light branches (persistence on). The direct `setAttribute('theme', ...)` and `localStorage.setItem(...)` calls are removed — `app.setTheme()` owns persistence end-to-end.

The plan also adds a regression test in `src/app.test.ts` that exercises the new propagation path AND a sibling assertion covering the 'System' branch's localStorage contract. The tests fail on `main` (pre-fix) and pass after the fix lands.

Purpose: Restore correct behavior on the Settings → navbar theme flow. Close CR-01 WITHOUT regressing the 'System' theme UX contract. Lock in both invariants with automated tests so neither class of bug can reappear.

## Architectural Choice: Option B (IN-04)

Two options surfaced by the code reviewer and verifier (verifier preference: B):

- **Option A** — Make `settings.ts setTheme()` go through `<zt-app>.toggleTheme()` directly. KEEPS the navbar's localStorage-on-connect read AS-IS. Relies on `toggleTheme()` re-syncing the navbar by side-effect. **Insufficient alone** because (a) `toggleTheme()` toggles unconditionally — settings has 3 targets (dark/light/system), not 2; (b) even after `app.theme` changes, the navbar has no observer for that change unless we also add a prop binding. Option A alone leaves the navbar stale in the system-detect → light-on-light-display case.
- **Option B** — Thread `<zt-app>.theme` to `<zt-navbar>.currentTheme` as a Lit property binding; drop navbar's localStorage read; add `app.setTheme(target, options?)` for callers. **Makes CR-01 impossible by construction** (verifier's words). Single source of truth: `<zt-app>.theme @state`.

**Choice: Option B.** Rationale:
1. **Single source of truth.** Lit's reactive property system is the right mechanism for parent → child theme flow. The localStorage read in navbar's `connectedCallback` is a side-effect-as-state-management anti-pattern that masked the bug before Phase 20.
2. **No timing fragility.** With Option A, the order of `toggleTheme()` side-effects matters (does it update localStorage before the navbar's next render? does it dispatch an event?). With Option B, Lit handles re-render on `@state` change automatically.
3. **Cleaner API surface.** `<zt-app>.setTheme(target, options?)` is a proper public method for explicit theme setting (used by settings); `<zt-app>.toggleTheme()` remains the toggle convenience (used by the navbar's theme button). Both delegate to the same `@state` mutation path.
4. **The verifier preferred B**, the code reviewer's IN-04 entry recommended B, and B closes both CR-01 and the underlying IN-04 design issue in one pass.

**Why not Option A:** Even paired with a prop binding, Option A's `toggleTheme()` does not accept a target — it only flips. To handle settings' three buttons (dark/light/system), the call site would need to compute the current theme, compare to the target, and conditionally call `toggleTheme()`. That conditional logic is fragile (off-by-one toggle if state and attribute drift), and it leaves the public API asymmetric (toggle but no set). Option B replaces this with a clean `setTheme(target, options?)` method.

## 'System' Theme UX Contract (preserved by `options.persist`)

The pre-Phase-20 'System' contract — still active in `src/app.ts:71-77` `firstUpdated()` — is:

- `localStorage.getItem('zt-theme') === null` → boot reads `window.matchMedia('(prefers-color-scheme: light)')` → app follows live OS preference on every page load.

A naive Option B implementation would break this contract: if the Settings 'System' branch calls `app.setTheme(resolved)` with the default `persist: true`, then `localStorage` always carries a `zt-theme` value after any Settings click, the `matchMedia` fallback in `firstUpdated` becomes effectively dead code, and tomorrow's OS theme change is ignored at boot. That is a side-regression — exactly the scope-reduction anti-pattern this plan must avoid.

**Fix (B-1 from plan-checker):** `<zt-app>.setTheme()` accepts an `options: { persist?: boolean }` second parameter. The Settings 'System' branch calls `app.setTheme(resolved, { persist: false })` so the live UI updates (theme @state mutates, navbar re-renders) WITHOUT writing localStorage; then settings.ts explicitly `localStorage.removeItem('zt-theme')` to guarantee the key is absent for the next-boot OS-preference re-read. The dark/light branches keep the default `persist: true` so explicit user choice is remembered.

This preserves the System UX contract end-to-end: clicking System resolves the live theme AND clears localStorage so next boot re-reads OS preference.

## What Does NOT Change

- `src/components/navbar.ts`'s `handleThemeToggle()` continues to route through `app.toggleTheme()` (the navbar's own theme button). The only change inside `handleThemeToggle()` is removing the line `this.currentTheme = app.currentTheme;` — that assignment is now unnecessary because the parent's re-render will update the property via the binding.
- `src/components/navbar.ts`'s `Sun` / `Moon` SVG selection at line 187, and the aria-label / title at lines 184-185, continue to read `this.currentTheme` — but now `currentTheme` is a `@property` fed by the parent, so reads remain identical.
- `src/styles/theme.ts` is NOT modified. The theme tokens, `data-theme` attribute, and MIRROR-fenced literal block are untouched.
- `src/app.ts`'s `firstUpdated()` localStorage read (lines 71-77) stays AS-IS — that is the BOOT-TIME read that seeds `<zt-app>.theme` from prior session (and falls back to `matchMedia` when the key is absent). Removing it would lose persisted theme across full-page reloads AND break the 'System' contract.
- `src/pages/settings.ts`'s `connectedCallback` (which reads localStorage to seed the settings page's own `this.currentTheme` for the "active button highlight") stays AS-IS — that read is purely UI-state for the settings page's three-button selector and does not need to be the source of truth.

- Existing `src/components/navbar.test.ts` tests (8 tests) continue to pass. They stub localStorage.getItem to return 'dark' (line 9), which previously fed `connectedCallback`'s read. After this plan, that read is gone — but the stub is harmless (no consumer left).
