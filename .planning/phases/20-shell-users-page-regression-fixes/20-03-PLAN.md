**Why event.detail, not Router.location:** In Vaadin Router 2.x the `location` is an **instance property** on a Router instance, NOT a static property on the `Router` class. Reading `Router.location` (static access on the class) compiles in TypeScript (Vaadin's types do not flag the misuse) but returns `undefined` at runtime — the navbar would render with empty title/subtitle on every authenticated route and the bug would ship silently. The `vaadin-router-location-changed` event carries the full resolved location at `event.detail.location`, and the route metadata lives at `event.detail.location.route` (Plan 20-02 added the `title`/`subtitle` fields to each route definition). Reading from the event detail is the canonical Vaadin Router 2.x pattern, requires no `Router` import, and stays type-safe via a defensive cast on the `route` shape.

**Vaadin Router 2.0.1 event-detail shape (the canonical wiring path):**
The `vaadin-router-location-changed` CustomEvent has the shape:
```ts
interface VaadinRouterLocationChangedEvent extends CustomEvent {
    detail: {
        router: Router;       // The Router instance that emitted the event
        location: RouterLocation;  // Includes .pathname, .params, .route
    };
}
```
`event.detail.location.route` resolves to the matched route object — including any custom fields declared on the route definition. Plan 20-02 added `title` and `subtitle` fields; they appear on `event.detail.location.route?.title` and `event.detail.location.route?.subtitle`. The `Route` type does NOT formally declare these custom fields; use a defensive cast inside the handler (see action below).

**What NOT to do (the BLOCKER pattern):** Do NOT write `Router.location?.route` — in Vaadin Router 2.x `location` is an instance property, not static. TypeScript will silently accept the access (returns `undefined`) and the navbar will render with empty title/subtitle on every authenticated route, shipping the bug.
